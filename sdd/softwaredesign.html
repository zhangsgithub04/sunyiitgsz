01

High-Level Design (HLD)

The "What" and the "Global Context"

High-Level Design focuses on the conceptual landscape. It defines system boundaries, user interactions, and external dependencies. This is where we decide how the application fits into the world.

  • Context Diagrams: Identifying external users and third-party APIs (like Auth0 or AWS S3).
  • User Journeys: Mapping the path from landing page to collaborative drawing.

Whiteboard Application Context

In HLD, we establish that our Whiteboard system consists of a Web Client, a Synchronization Cluster, and a Storage Engine. We define that users must be able to share a "Board URL" and see peers' cursors in real-time.

02

Architectural Design

The "Skeleton" and "Decision Making"

Patterns

Choosing between Monolithic, Microservices, or Serverless. For our whiteboard, we select Event-Driven Architecture to handle live WebSocket streams.

Data Strategy

Deciding on Redis for ephemeral real-time state (cursor positions) and PostgreSQL for persistent board data (saved shapes).

Quality Attributes

Defining "The -ilities": Scalability to 10k users, Availability of 99.9%, and sub-50ms Latency for ink synchronization.
